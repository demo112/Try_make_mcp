Model Context Protocol (MCP) servers can fail for many reasons — misconfigured endpoints, expired tokens, transport mismatches, or protocol version incompatibilities. This guide covers the 40+ most common MCP errors with practical fixes you can apply immediately.

1. Connection Refused / Cannot Connect

Error: ECONNREFUSED or Connection refused

Error: connect ECONNREFUSED 127.0.0.1:8080
Error: Connection refused — is the MCP server running?

Causes & Fixes:

✅ Fix 1: Verify the server is running
MCP servers using HTTP/SSE transport must be actively listening before the client connects. Start the server first:
python -m mcp_server my_server.py
# Or with uvicorn for FastAPI-based servers:
uvicorn my_server:app --host 127.0.0.1 --port 8080
✅ Fix 2: Check the correct port and host
Misconfigured host/port is the #1 cause. Verify in your client config:
# Claude Desktop config (~/.claude_desktop_config.json)
{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["/path/to/server.py"],
      "env": {
        "MCP_PORT": "8080",
        "MCP_HOST": "127.0.0.1"
      }
    }
  }
}
✅ Fix 3: Check firewall / port availability
# macOS/Linux: Check if port is listening
lsof -i :8080
netstat -tlnp | grep 8080

# Test connectivity
curl http://127.0.0.1:8080/health
✅ Fix 4: Docker port mapping
If running inside Docker, ensure ports are mapped correctly:
docker run -p 127.0.0.1:8080:8080 my-mcp-server

Error: ENOTFOUND or DNS resolution failure

Error: getaddrinfo ENOTFOUND api.example.com Error: DNS lookup failed for remote MCP server
✅ Fix: Check DNS and network connectivity
# Verify DNS resolution
nslookup api.example.com
dig api.example.com

# Test with curl
curl -v https://api.example.com/mcp

# Check proxy settings (common in corporate environments)
echo $HTTP_PROXY
echo $HTTPS_PROXY

2. Authentication & Authorization Failures

Error: 401 Unauthorized or Authentication failed

Error: 401 Unauthorized — Invalid or missing API key
Error: Authentication failed: Bearer token expired
Error: HMAC signature verification failed
✅ Fix 1: Verify API key format and placement
Different transports require different key placements:
# For stdio transport: key in process environment
{
  "command": "python",
  "args": ["/path/to/server.py"],
  "env": {
    "API_KEY": "sk-xxxxxxxxxxxxxxxx"
  }
}

# For HTTP/SSE: key in Authorization header
curl -H "Authorization: Bearer sk-xxx" https://api.example.com/mcp

# For API key in header
curl -H "X-API-Key: sk-xxx" https://api.example.com/mcp
✅ Fix 2: Check token expiration
Many MCP providers use short-lived tokens. Implement refresh logic:
import time

class TokenManager:
    def __init__(self, api_key):
        self.api_key = api_key
        self.expires_at = time.time() + 3600  # 1 hour
    
    def get_valid_token(self):
        if time.time() >= self.expires_at:
            self.api_key = self.refresh_token()
            self.expires_at = time.time() + 3600
        return self.api_key
✅ Fix 3: OAuth 2.0 configuration errors
Common OAuth mistakes with MCP servers:
# Wrong: Token in query param (some servers reject this)
curl "https://api.example.com/mcp?token=sk-xxx"  # ❌

# Correct: Authorization header
curl -H "Authorization: Bearer sk-xxx" https://api.example.com/mcp

# Correct: mTLS with client certificates
curl --cert client.crt --key client.key https://api.example.com/mcp

3. Timeout & Latency Issues

Error: TimeoutError or Request timeout after N ms

Error: TimeoutError: Request to MCP server timed out after 30000ms
Error: Server did not respond within configured timeout (5000ms)
Error: Tool execution exceeded maximum allowed duration
Timeout TypeDefault ValueWhen to Increase
Connection timeout5-10 secondsRemote servers, cold starts
Request timeout30 secondsLong-running tool calls, RAG
Tool call timeoutVaries (10s-5min)File I/O, database queries
Streaming timeout60 secondsLarge response streaming
✅ Fix 1: Configure timeouts appropriately
# Claude Desktop: configure timeouts in config
{
  "mcpServers": {
    "my-server": {
      "command": "python",
      "args": ["/path/to/server.py"],
      "timeout": 120,  // seconds, for all requests
      "env": {
        "REQUEST_TIMEOUT": "60",
        "CONNECT_TIMEOUT": "30"
      }
    }
  }
}
✅ Fix 2: Implement proper streaming for long responses
Long responses should use Server-Sent Events (SSE) streaming to avoid timeouts:
@app.post("/mcp")
async def mcp_endpoint(request: Request):
    async def event_generator():
        result = await process_long_task(request)
        # Stream results instead of waiting
        for chunk in result:
            yield f"data: {json.dumps(chunk)}\n\n"
    
    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream"
    )
✅ Fix 3: Cold start optimization for containerized servers
# Dockerfile with pre-warming
FROM python:3.11-slim
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Pre-warm: import all modules at build time
RUN python -c "import mcp_server; import handlers; import db"

COPY . .
CMD ["python", "-m", "uvicorn", "app:app", "--host", "0.0.0.0"]

4. Transport Protocol Errors

Error: Invalid transport type or Unsupported transport

Error: Invalid transport type 'http'. Supported: stdio, sse
Error: Transport mismatch: client uses stdio, server expects HTTP/SSE
✅ Fix 1: Know which transport each MCP server requires
MCP officially supports two transports:

  • stdio: Spawns server as subprocess. Used by Claude Desktop, Cursor, VS Code extensions.
  • HTTP/SSE: Server runs as network service. Used by remote MCP servers, web integrations.
# stdio transport (for Claude Desktop, etc.)
{
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"]
}

# HTTP/SSE transport (for remote servers)
{
  "url": "https://api.example.com/mcp/sse"
}
✅ Fix 2: Transport mismatch between client and server
If your client uses stdio but server exposes HTTP:
# Option A: Use an MCP proxy that bridges stdio <-> HTTP
npx @modelcontextprotocol/server-http-proxy \
  --port 3000 \
  --http-endpoint https://api.example.com/mcp

# Option B: Use Try_make_mcp to create a stdio-compatible wrapper
python src/mcp_factory/run.py --mode stdio --target http://api.example.com/mcp

5. Stdio Transport Common Issues

Error: Process exited with code 1 or Server process died

Error: MCP server process exited with code 1
Error: Stderr: ModuleNotFoundError: No module named 'mcp'
Error: Stderr: ImportError: libffi.so.7: cannot open shared object file
✅ Fix 1: Missing Python dependencies
# Install the required MCP Python package
pip install mcp

# Or for servers requiring specific versions
pip install "mcp>=1.0.0"

# Check what's installed
pip list | grep -i mcp
✅ Fix 2: System library missing (libffi on Linux/macOS)
# macOS
brew install libffi
export PKG_CONFIG_PATH="/usr/local/opt/libffi/lib/pkgconfig"

# Ubuntu/Debian
sudo apt-get install libffi-dev python3-dev

# Then reinstall Python packages
pip install --force-reinstall cffi
✅ Fix 3: Wrong Python version
# MCP servers may require specific Python versions
python --version  # Should be 3.9+

# If using virtualenv, ensure it's activated
source venv/bin/activate
python --version

6. JSON-RPC Protocol Errors

Error: Invalid JSON-RPC request or Parse error

Error: Parse error: Unexpected token at position 0
Error: Invalid JSON-RPC response: missing 'jsonrpc' field
Error: Method not found: 'tools/call' is not a valid MCP method
✅ Fix 1: Verify JSON-RPC 2.0 compliance
MCP uses JSON-RPC 2.0. Every request MUST include:
# Valid MCP request
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}

# MCP responses must include jsonrpc and id
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [...]
  }
}

# Error responses
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32600,
    "message": "Invalid Request"
  }
}
✅ Fix 2: Notification vs Request distinction
Notifications (no id) are one-way and the server must NOT respond:
# This is a notification — NO response expected
{"jsonrpc": "2.0", "method": "notifications/initialized"}

# This is a request — MUST respond
{"jsonrpc": "2.0", "id": 1, "method": "tools/list"}
✅ Fix 3: Protocol method name errors
MCP defines specific method names. Common mistakes:
# Correct MCP method names:
"initialize"          # First call to establish protocol
"tools/list"          # List available tools
"tools/call"          # Call a tool
"resources/list"      # List resources
"resources/read"      # Read a resource
"prompts/list"        # List prompts
"sampling/createMessage"  # Request LLM sampling

# WRONG (MCP doesn't use these names):
"mcp_tools_list"      # ❌
"list_tools"          # ❌
"/tools/list"         # ❌

7. MCP Manifest / Configuration Errors

Error: Invalid server manifest or Schema validation failed

Error: Server manifest validation failed: missing 'name' field
Error: Invalid capabilities object in server manifest
Error: Protocol version mismatch: server supports 2024-11-05, client supports 2024-12-06
✅ Fix 1: Complete and valid server manifest
The MCP initialization handshake requires this exact structure:
{
  "protocolVersion": "2024-11-05",
  "capabilities": {
    "tools": {},
    "resources": {},
    "prompts": {}
  },
  "serverInfo": {
    "name": "my-mcp-server",
    "version": "1.0.0"
  }
}
✅ Fix 2: Protocol version negotiation
Client and server must agree on a protocol version:
# Client sends its supported versions in initialize:
{
  "protocolVersion": "2024-12-06",
  "capabilities": { "roots": { "listChanged": true } },
  "clientInfo": { "name": "claude-desktop", "version": "1.0" }
}

# Server responds with the version it will use:
{
  "protocolVersion": "2024-11-05",  # Common version both support
  "capabilities": { "tools": {} },
  "serverInfo": { "name": "my-server", "version": "1.0" }
}
✅ Fix 3: Capabilities mismatch
If server declares capabilities it doesn't actually implement:
# Only declare what you actually implement:
"capabilities": {
  "tools": {},        # Only if you implement tools/list and tools/call
  "resources": {},    # Only if you implement resources/list and resources/read
  "prompts": {},      # Only if you implement prompts/list
  "logging": {}       # Only if you implement logging/message
}

8. Resource & Tool Discovery Failures

Error: Unknown tool or Tool not found

Error: Unknown tool 'database_query'. Available tools: ['search', 'fetch']
Error: Tool 'files/search' not found in server capabilities
Error: Resource not found: 'config://app/settings'
✅ Fix 1: Always query tools/list first
Before calling any tool, always discover what's available:
# Correct workflow:
1. Send: {"jsonrpc": "2.0", "id": 1, "method": "tools/list", "params": {}}
2. Receive: {"jsonrpc": "2.0", "id": 1, "result": {"tools": [...]}}
3. Pick tool by name from the list
4. Call with exact name from the manifest

# Wrong: hardcoding tool names without discovery
{"method": "tools/call", "params": {"name": "search"}}  # ❌ guessing
✅ Fix 2: Handle optional capabilities gracefully
Not all MCP servers implement all features:
# Check server capabilities before calling
if "tools" in server_capabilities:
    tools = await list_tools()
    
if "resources" in server_capabilities:
    resources = await list_resources()
    
# For optional features, handle NotImplemented gracefully
try:
    result = await call_tool("optional_feature")
except MethodNotFoundError:
    print("Tool not available on this server")

9. Sampling / SamplingRequest Errors

Error: SamplingRequest rejected or Invalid sampling parameters

Error: SamplingRequest: model 'gpt-4' not allowed by server policy
Error: Invalid sampling request: missing 'systemPrompt' field
Error: Sampling rejected: context window exceeded (200k > 128k limit)
✅ Fix: Implement proper sampling request handling
# Sampling is used when the server needs the LLM to generate text
# It's an advanced feature that requires careful implementation

@sse.on_message("sampling/createMessage")
async def handle_sampling_request(request: SamplingRequest):
    # Validate request parameters
    if not request.prompt:
        raise ValueError("Missing required field: prompt")
    
    if request.maxTokens > MAX_SUPPORTED_TOKENS:
        raise ValueError(f"maxTokens exceeds limit: {MAX_SUPPORTED_TOKENS}")
    
    # Implement allowlist policy for models
    allowed_models = ["claude-3-5-sonnet", "gpt-4o", "gpt-4-turbo"]
    if request.model not in allowed_models:
        raise ValueError(f"Model '{request.model}' not in allowlist")
    
    # Return the sampling response
    return {
        "model": request.model,
        "completion": await llm.generate(request.prompt),
        "stopReason": "endTurn"
    }

10. Debugging Tools & Techniques

Enable verbose logging

✅ Debug technique 1: Enable MCP SDK debug logging
import logging

# Enable all MCP debug logs
logging.basicConfig(level=logging.DEBUG)

# For MCP Python SDK specifically
import mcp.server.stdio
logging.getLogger("mcp.server").setLevel(logging.DEBUG)
logging.getLogger("mcp.protocol").setLevel(logging.DEBUG)
logging.getLogger("mcp.transport").setLevel(logging.DEBUG)
✅ Debug technique 2: Use MCP CLI inspector
The official MCP CLI provides a debugging inspector:
# Install MCP CLI
npm install -g @modelcontextprotocol/cli

# Run the inspector to test your server
mcp inspect stdio --command "python" --args "/path/to/server.py"

# Test HTTP/SSE servers
mcp inspect http --url http://localhost:8080/mcp/sse
✅ Debug technique 3: Wire-level protocol logging
For stdio transport, capture the raw JSON-RPC messages:
# Wrap your server to log all incoming/outgoing messages
class LoggingStdioServer:
    def __init__(self, server):
        self.server = server
    
    def handle_message(self, msg):
        print(f"[IN]  {json.dumps(msg)}", file=sys.stderr)
        response = self.server.process(msg)
        print(f"[OUT] {json.dumps(response)}", file=sys.stderr)
        return response
✅ Debug technique 4: Test with Try_make_mcp validation
Use Try_make_mcp's built-in MCP validator to test compliance:
# Run Try_make_mcp's validation suite against your server
python -m mcp_factory validate \

  --command "python" \
  --args "server.py"

# Check for protocol compliance issues
python -m mcp_factory test-compliance \
  --server-url http://localhost:8080/mcp/sse
