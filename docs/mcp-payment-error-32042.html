MCP Error -32042: The Complete Guide to Payment Required in MCP Servers

MCP Spec Production Pattern Freemium Model

10 min read · Updated May 2026 · Covers MCP Discussion #2436, BGPT MCP, and implementation best practices

Error code -32042 is the MCP ecosystem's standard signal for "Payment Required". This guide covers everything you need to know: what it is, how to implement it, how clients handle it, and how to build a complete freemium monetization layer around it.

What is Error -32042?

JSON-RPC reserves error codes in the -32000 to -32099 range for "Server Errors" that are application-defined. The MCP community proposed -32042 specifically for payment-required scenarios in MCP Discussion #2436.

Origin: Proposed in MCP Specification Discussion #2436 as a standard way for MCP servers to signal that a payment is needed before processing a request. This follows the HTTP 402 Payment Required pattern adapted for the MCP/JSON-RPC protocol.

Complete Implementation Guide

Architecture: The Freemium Gate

Here's how a production MCP server handles the payment flow:

1
Request arrives — MCP client sends a tool call (e.g., tools/call)
2
Check API key — Look for api_key in tool arguments. If valid → execute immediately ✅
3
Check free tier — No API key? Check IP-based rate limit. If under limit → execute and increment counter ✅
4
Return -32042 — Free tier exhausted AND no API key? Return payment required error with subscribe URL ❌

Implementation: Cloudflare Worker (Production Code)

The following is adapted from our Stripe MCP Integration Tutorial and runs on Cloudflare Workers with KV persistence:

// Validate API key against environment whitelist
function validateApiKey(apiKey, env) {
  if (!apiKey) return false;
  const validKeys = (env.VALID_API_KEYS || '').split(',').filter(Boolean);
  return validKeys.includes(apiKey);
}

// Check IP-based free tier with KV persistence
async function checkFreeTierLimit(ip, env) {
  const today = new Date().toISOString().split('T')[0];
  const key = `usage:${ip}:${today}`;
  
  // Try KV first (persistent across requests)
  if (env.MCP_USAGE) {
    const count = parseInt(await env.MCP_USAGE.get(key) || '0');
    if (count >= 10) return false;
    await env.MCP_USAGE.put(key, String(count + 1));
    return true;
  }
  
  // Fallback: in-memory (single-instance only)
  if (!globalThis.usageCounts) globalThis.usageCounts = {};
  const count = globalThis.usageCounts[key] || 0;
  if (count >= 10) return false;
  globalThis.usageCounts[key] = count + 1;
  return true;
}

// Build the -32042 error response
function paymentRequiredError(subscribeUrl, pricingUrl) {
  return {
    jsonrpc: '2.0',
    id: null,
    error: {
      code: -32042,
      message: 'Payment required: Free tier exhausted',
      data: {
        subscribe_url: subscribeUrl,
        pricing_url: pricingUrl,
        free_tier_remaining: 0,
        upgrade_message: 'You have used all 10 free requests for today. ' +
          'Upgrade for unlimited access at ' + subscribeUrl
      }
    }
  };
}

Integration: Wire It Into Your Request Handler

async function handleToolCall(request, env) {
  const body = await request.json();
  const toolArgs = body.params?.arguments || {};
  const clientIP = request.headers.get('CF-Connecting-IP') || 'unknown';
  
  // Step 1: Check for valid API key
  if (validateApiKey(toolArgs.api_key, env)) {
    // Strip api_key from args before passing to tool
    const { api_key, ...cleanArgs } = toolArgs;
    return executeTool(body.params.name, cleanArgs);
  }
  
  // Step 2: Fall back to free tier
  const allowed = await checkFreeTierLimit(clientIP, env);
  if (allowed) {
    return executeTool(body.params.name, toolArgs);
  }
  
  // Step 3: Payment required
  return paymentRequiredError(
    `${env.BASE_URL || 'https://your-domain.com'}/subscribe`,
    `${env.BASE_URL || 'https://your-domain.com'}/pricing`
  );
}

Client-Side Error Handling

If you're building an MCP client, here's how to handle -32042 gracefully:

// Client-side error handler (TypeScript)
function handleMCPError(error: MCPError) {
  switch (error.code) {
    case -32042: {
      // Payment required — extract upgrade info
      const data = error.data;
      console.log('Upgrade needed:', data.upgrade_message);
      console.log('Subscribe at:', data.subscribe_url);
      console.log('Pricing info:', data.pricing_url);
      
      // Show user-friendly upgrade prompt
      showUpgradeDialog({
        message: data.upgrade_message,
        subscribeUrl: data.subscribe_url,
        pricingUrl: data.pricing_url
      });
      break;
    }
    case -32001: {
      console.error('Invalid API key. Please check your credentials.');
      break;
    }
    default: {
      console.error(`MCP Error ${error.code}: ${error.message}`);
    }
  }
}

Best Practices for -32042

✅ Do: Provide Rich Error Data

Good — Informative 
{
  "code": -32042,
  "message": "Payment required",
  "data": {
    "subscribe_url": "https://...",
    "pricing_url": "https://...",
    "free_tier_remaining": 0,
    "upgrade_message": "Upgrade..."
  }
}
Bad — Bare minimum 
{
  "code": -32042,
  "message": "Pay up"
}

✅ Do: Track Usage Per Day, Not Per Request

Reset free tier counters daily. Use ISO date strings as KV keys (e.g., usage:192.168.1.1:2026-05-13) to ensure counters reset at midnight UTC.

✅ Do: Use CF-Connecting-IP on Cloudflare Workers

The standard X-Forwarded-For header can be spoofed. Cloudflare Workers provide CF-Connecting-IP which is always the real client IP.

❌ Don't: Block Before Showing Value

Anti-pattern: Never require payment before the user has experienced any value. Always provide a generous free tier (10-50 calls/day) so users can evaluate your tool before hitting the paywall.

❌ Don't: Return -32042 for Auth Errors

Use -32001 for invalid/expired API keys and -32042 only when payment is specifically required. Confusing these signals makes client-side handling unreliable.

Quick Start Checklist

Get -32042 working in your MCP server in 4 steps:
  1. Add API key validation — Check api_key param against env whitelist
  2. Add IP rate limiting — Count requests per IP per day using KV/Redis
  3. Return -32042 — When free tier is exhausted, include subscribe URL in error data
  4. Set up payment — Create Stripe Checkout session that generates API keys on success

Total time: ~45 minutes if you already have a working MCP server.
