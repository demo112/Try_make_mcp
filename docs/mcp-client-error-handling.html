1. Complete JSON-RPC 2.0 Error Code Reference

Every MCP server uses JSON-RPC 2.0 under the hood. These standard error codes apply to all MCP implementations:

CodeNameMCP MeaningRecovery
-32700 Parse Error Invalid JSON in request Log payload, fix serialization
-32600 Invalid Request Malformed request object (missing id/method/params) Validate request schema before sending
-32601 Method Not Found Tool or resource name doesn't exist on server Check server capabilities via list_tools()
-32602 Invalid Params Tool arguments don't match server schema Validate params against tool schema before call
-32603 Internal Error Server crashed or unhandled exception Retry with backoff, alert server operator

2. MCP-Specific Error Codes

Beyond standard JSON-RPC, the MCP ecosystem uses custom error codes in the -32000 to -32099 range:

CodeNameSeverityWhen It HappensWhat To Do
-32001 Server Not Initialized Critical Calling tools before initialize handshake Send initialize request first
-32002 Unknown Method Warning Server doesn't support requested method Refresh server capabilities
-32042 Payment Required Critical Free tier exhausted, API key invalid/missing Redirect to subscribe URL in error data
-32043 Rate Limited Warning Too many requests in time window Wait for reset time, implement backoff
-32044 Resource Not Found Warning Requested URI doesn't exist on server Update resource list, handle gracefully
-32045 Auth Required Critical Server requires authentication Provide API key or OAuth token
💡 Key Insight: Custom error codes have no universal meaning across MCP servers. Always read the error.data field for server-specific recovery instructions. A well-designed MCP server includes a recoveryHint and/or subscribeUrl in the error data.

3. Connection Failure Patterns

Before you even get JSON-RPC errors, the connection itself can fail. Here are the 5 most common patterns:

Pattern 1: Server Startup Race Condition

The client connects before the MCP server has finished initializing. This is the most common error in stdio-based MCP servers (Claude Desktop, Cursor).

# What happens:
Client → stdio: {"jsonrpc":"2.0","id":1,"method":"initialize",...}
Server → stderr: "Loading plugin..."
Client ← (timeout after 5s, no response)
Server → stderr: "Plugin loaded."
Server → stdout: {"jsonrpc":"2.0","id":1,"result":{...}}  # Too late!

Fix: Implement a readiness check. Wait for the server's first stdout message before sending initialize.

Pattern 2: SSE Connection Drops (HTTP Transport)

Server-Sent Events connections silently drop after 30-60 seconds of inactivity. Many Cloudflare-based MCP servers hit this.

# Symptom: Connection appears open but no responses
# Fix: Send periodic heartbeat or re-connect on 15s idle
setInterval(() => {
  if (Date.now() - lastMessage > 15000) reconnect();
}, 5000);

Pattern 3: Streamable HTTP Transport Errors

The newer Streamable HTTP transport (MCP spec 2025-03) uses POST for all communication. Failed POSTs return HTTP status codes:

HTTP StatusMCP MeaningAction
200SuccessProcess response
202Accepted (streaming)Read SSE stream
400Bad RequestFix JSON-RPC format
401UnauthorizedAdd auth header
402Payment RequiredCheck API key / subscribe
405Method Not AllowedServer doesn't support this transport
408Request TimeoutRetry with exponential backoff
500Server ErrorRetry, then fail gracefully

4. Timeout & Retry Strategies

Not all errors should be retried. Here's when to retry and when to fail fast:

✅ Safe to Retry (with backoff):
  • -32603 Internal Error (server may have recovered)
  • -32043 Rate Limited (wait for reset time)
  • Connection timeout / network error
  • HTTP 408 Request Timeout
  • HTTP 500 Server Error
❌ Do NOT Retry (fail immediately):
  • -32600 Invalid Request (your bug, retrying won't help)
  • -32601 Method Not Found (server capability mismatch)
  • -32602 Invalid Params (fix your arguments)
  • -32001 Server Not Initialized (fix your handshake order)
  • -32045 Auth Required (retrying without credentials is useless)
  • -32042 Payment Required (retrying won't give you money)
  • HTTP 400 Bad Request, HTTP 401 Unauthorized

Exponential Backoff Implementation

function withRetry<T>(
  fn: () => Promise<T>,
  maxRetries = 3,
  baseDelay = 1000
): Promise<T> {
  return fn().catch(err => {
    const code = err?.jsonrpc?.error?.code;
    // Only retry retryable errors
    if (maxRetries === 0 || !isRetryable(code, err?.statusCode)) {
      throw err;
    }
    const delay = baseDelay * Math.pow(2, maxRetries - 1) + Math.random() * 500;
    return new Promise(resolve => setTimeout(resolve, delay))
      .then(() => withRetry(fn, maxRetries - 1, baseDelay));
  });
}

function isRetryable(code: number, httpStatus?: number): boolean {
  const retryableCodes = [-32603, -32043];
  const retryableHttp = [408, 500, 502, 503, 504];
  if (retryableCodes.includes(code)) return true;
  if (httpStatus && retryableHttp.includes(httpStatus)) return true;
  if (!code && !httpStatus) return true; // Network error
  return false;
}

5. Payment & Rate Limit Errors (-32042)

As MCP servers adopt freemium models (BGPT MCP, APIbase.pro), the -32042 Payment Required error is becoming critical. Here's how to handle it properly:

// Error structure from a freemium MCP server:
{
  "jsonrpc": "2.0",
  "id": 42,
  "error": {
    "code": -32042,
    "message": "Payment required: daily free limit exceeded",
    "data": {
      "freeLimit": 10,
      "usedToday": 10,
      "resetAt": "2026-05-14T00:00:00Z",
      "subscribeUrl": "https://checkout.stripe.com/pay-link#...",
      "recoveryHint": "Add your API key to the 'api_key' tool parameter, or visit subscribeUrl"
    }
  }
}

Best Practice: Graceful Degradation

async function callToolWithPaymentHandling(server, toolName, args) {
  try {
    return await server.callTool(toolName, args);
  } catch (err) {
    if (err.code === -32042) {
      // 1. Log for analytics
      logger.warn('MCP payment required', { server: server.name, tool: toolName });

      // 2. Check if user has API key configured
      if (!config.getApiKey(server.name)) {
        return {
          content: [{
            type: "text",
            text: `⚠️ Free limit reached for ${server.name}.\n\n` +
                  `Options:\n` +
                  `1. Get an API key: ${err.data.subscribeUrl}\n` +
                  `2. Wait for reset: ${new Date(err.data.resetAt).toLocaleString()}\n` +
                  `3. Try again tomorrow (${err.data.freeLimit} free calls/day)`
          }],
          isError: true
        };
      }

      // 3. If API key exists but failed, report configuration issue
      throw new McpConfigError(
        `API key invalid for ${server.name}. Check: ${err.data.subscribeUrl}`,
        { subscribeUrl: err.data.subscribeUrl }
      );
    }
    throw err;
  }
}
⚠️ Critical: Never silently swallow -32042 errors. The user needs to know the tool requires payment. Always surface the subscribeUrl and resetAt from error data.

6. Production TypeScript Implementation

A complete, typed error handler for any MCP client:

import type { JSONRPCError, McpErrorData } from './types';

enum McpErrorCode {
  SERVER_NOT_INITIALIZED = -32001,
  UNKNOWN_METHOD = -32002,
  PAYMENT_REQUIRED = -32042,
  RATE_LIMITED = -32043,
  RESOURCE_NOT_FOUND = -32044,
  AUTH_REQUIRED = -32045,
}

type ErrorSeverity = 'critical' | 'recoverable' | 'warn' | 'info';

interface ErrorStrategy {
  severity: ErrorSeverity;
  retryable: boolean;
  userMessage: string;
  action?: string;
}

const ERROR_STRATEGIES: Record<number, ErrorStrategy> = {
  [-32700]: { severity: 'critical', retryable: false,
    userMessage: 'Failed to communicate with MCP server.', action: 'Check server logs.' },
  [-32600]: { severity: 'critical', retryable: false,
    userMessage: 'Invalid request sent to MCP server.', action: 'Report this bug.' },
  [-32601]: { severity: 'warn', retryable: false,
    userMessage: 'Tool not available on this server.', action: 'Run list_tools() to see available tools.' },
  [-32602]: { severity: 'critical', retryable: false,
    userMessage: 'Invalid arguments for tool call.', action: 'Check tool schema.' },
  [-32603]: { severity: 'recoverable', retryable: true,
    userMessage: 'MCP server encountered an error.', action: 'Retrying...' },
  [McpErrorCode.SERVER_NOT_INITIALIZED]: { severity: 'critical', retryable: false,
    userMessage: 'MCP server not initialized.', action: 'Restart the connection.' },
  [McpErrorCode.PAYMENT_REQUIRED]: { severity: 'critical', retryable: false,
    userMessage: 'Payment required.', action: 'Check error.data.subscribeUrl' },
  [McpErrorCode.RATE_LIMITED]: { severity: 'recoverable', retryable: true,
    userMessage: 'Rate limited by MCP server.', action: 'Waiting and retrying...' },
  [McpErrorCode.AUTH_REQUIRED]: { severity: 'critical', retryable: false,
    userMessage: 'Authentication required.', action: 'Configure API key.' },
};

function handleMcpError(error: JSONRPCError): ErrorStrategy {
  const code = error.error?.code ?? error.statusCode ?? 0;
  const strategy = ERROR_STRATEGIES[code];

  if (strategy) {
    // Enrich payment errors with subscribe URL
    if (code === McpErrorCode.PAYMENT_REQUIRED && error.error?.data?.subscribeUrl) {
      strategy.action = `Subscribe at: ${error.error.data.subscribeUrl}`;
    }
    return strategy;
  }

  // Unknown error — be cautious
  return {
    severity: 'warn',
    retryable: false,
    userMessage: `Unknown MCP error: ${code}`,
    action: 'Check server documentation.',
  };
}

// Usage in your MCP client loop:
async function safeToolCall(server, toolName, args, retries = 3) {
  try {
    const result = await server.callTool(toolName, args);
    return result;
  } catch (error) {
    const strategy = handleMcpError(error);

    // Log with appropriate level
    const logger = strategy.severity === 'critical' ? console.error :
                   strategy.severity === 'warn' ? console.warn : console.info;
    logger(`[MCP ${strategy.severity}] ${strategy.userMessage}`, {
      code: error.error?.code,
      server: server.name,
      tool: toolName,
    });

    // Retry or fail
    if (strategy.retryable && retries > 0) {
      const delay = 1000 * Math.pow(2, 3 - retries);
      await new Promise(r => setTimeout(r, delay));
      return safeToolCall(server, toolName, args, retries - 1);
    }

    // Return error result for agent consumption
    return {
      content: [{ type: 'text', text: `${strategy.userMessage}\n${strategy.action || ''}` }],
      isError: true,
    };
  }
}

7. Production Python Implementation

from dataclasses import dataclass
from enum import Enum
import asyncio
import logging

class Severity(Enum):
    CRITICAL = "critical"
    RECOVERABLE = "recoverable"
    WARN = "warn"
    INFO = "info"

@dataclass
class ErrorStrategy:
    severity: Severity
    retryable: bool
    user_message: str
    action: str = ""

# Error strategy lookup
STRATEGIES = {
    -32700: ErrorStrategy(Severity.CRITICAL, False, "Parse error", "Check JSON encoding"),
    -32600: ErrorStrategy(Severity.CRITICAL, False, "Invalid request", "Fix request format"),
    -32601: ErrorStrategy(Severity.WARN, False, "Method not found", "Check available tools"),
    -32602: ErrorStrategy(Severity.CRITICAL, False, "Invalid params", "Validate arguments"),
    -32603: ErrorStrategy(Severity.RECOVERABLE, True, "Server error", "Retrying..."),
    -32001: ErrorStrategy(Severity.CRITICAL, False, "Not initialized", "Restart connection"),
    -32042: ErrorStrategy(Severity.CRITICAL, False, "Payment required", "Check subscribe URL"),
    -32043: ErrorStrategy(Severity.RECOVERABLE, True, "Rate limited", "Wait and retry"),
    -32045: ErrorStrategy(Severity.CRITICAL, False, "Auth required", "Configure API key"),
}

async def safe_tool_call(server, tool_name: str, args: dict, max_retries: int = 3):
    """Call an MCP tool with full error handling and retry logic."""
    for attempt in range(max_retries + 1):
        try:
            return await server.call_tool(tool_name, args)
        except McpError as e:
            strategy = STRATEGIES.get(e.code)

            if strategy is None:
                logging.warning(f"Unknown MCP error: {e.code}")
                raise

            logging.log(
                logging.ERROR if strategy.severity == Severity.CRITICAL else logging.WARNING,
                f"[MCP {strategy.severity.value}] {strategy.user_message}"
            )

            if strategy.retryable and attempt < max_retries:
                delay = 1.0 * (2 ** attempt)
                logging.info(f"Retry {attempt + 1}/{max_retries} in {delay}s...")
                await asyncio.sleep(delay)
                continue

            # Payment required special handling
            if e.code == -32042 and e.data.get("subscribeUrl"):
                return {
                    "content": [{
                        "type": "text",
                        "text": f"Payment required.\nSubscribe: {e.data['subscribeUrl']}\n"
                                f"Free limit resets: {e.data.get('resetAt', 'unknown')}"
                    }],
                    "isError": True
                }

            raise McpHandledError(strategy.user_message, strategy.action)

    raise RuntimeError("Max retries exceeded")

10. Production Readiness Checklist

Before shipping your MCP client (or debugging someone else's), verify these items:

✅ Error Handling Checklist:
  1. ✅ Handle all 5 standard JSON-RPC error codes (-32700 to -32603)
  2. ✅ Handle MCP-specific codes (-32001, -32042, -32043, -32045)
  3. ✅ Implement exponential backoff for retryable errors only
  4. ✅ Surface error.data fields to end users (subscribeUrl, resetAt)
  5. ✅ Never retry client errors (-32600, -32602, -32042, -32045)
  6. ✅ Add connection timeout with automatic reconnection
  7. ✅ Validate tool params against server schema before calling
  8. ✅ Implement graceful degradation for payment errors
  9. ✅ Log all errors with severity level and context
  10. ✅ Handle HTTP-level errors (400, 401, 402, 408, 500)
  11. ✅ Test with a server that returns -32042 Payment Required
  12. ✅ Add health check / heartbeat for long-lived connections
❌ Anti-Patterns to Avoid:
  • ❌ Catching all errors with a generic catch(e) { console.log(e) }
  • ❌ Retrying -32602 Invalid Params (it's a client bug!)
  • ❌ Hiding payment errors from the user
  • ❌ Using fixed retry delays (always jitter!)
  • ❌ Ignoring error.data — it often contains recovery info
