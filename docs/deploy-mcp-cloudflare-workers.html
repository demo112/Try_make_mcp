Understanding MCP Transport: streamable-http

The MCP specification supports multiple transport mechanisms. The two most relevant for remote deployment are:

streamable-http (Recommended)

The modern transport, standardized in MCP 2025-03-26 protocol version. Clients send JSON-RPC messages via POST requests to a single endpoint:

# Client → Server
POST /mcp
Content-Type: application/json

{"jsonrpc":"2.0","id":1,"method":"initialize","params":{...}}

# Server → Client
HTTP 200 OK
Content-Type: application/json

{"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2025-03-26",...}}

sse (Legacy)

The original transport using Server-Sent Events. Still supported for backward compatibility, but streamable-http is recommended for all new deployments.

Key insight: 3,200+ servers in the MCP Registry use Remote mode. Cloudflare Workers natively supports both streamable-http and sse, making it the most accessible deployment option for MCP servers.

Step-by-Step Deployment

Step 1: Create the Worker

Create a new directory and initialize your Cloudflare Worker:

mkdir my-mcp-server && cd my-mcp-server
npm init -y
npm install wrangler -D
npx wrangler init

Step 2: Write the MCP Server Code

Here's a minimal but complete MCP server for Cloudflare Workers. It handles the full MCP initialization handshake, tool listing, and tool invocation:

// worker.mjs
export default {
  async fetch(request) {
    const url = new URL(request.url);

    // CORS
    if (request.method === 'OPTIONS') {
      return new Response(null, {
        status: 204,
        headers: {
          'Access-Control-Allow-Origin': '*',
          'Access-Control-Allow-Methods': 'GET, POST, OPTIONS',
          'Access-Control-Allow-Headers': 'Content-Type, Authorization',
        },
      });
    }

    // MCP endpoint
    if (url.pathname === '/mcp' && request.method === 'POST') {
      const body = await request.json();
      const { method, params, id } = body;

      let result, isError = false;

      switch (method) {
        case 'initialize':
          result = {
            protocolVersion: '2025-03-26',
            capabilities: { tools: { listChanged: false } },
            serverInfo: { name: 'my-mcp', version: '1.0.0' },
          };
          break;

        case 'notifications/initialized':
          return new Response(null, { status: 204 });

        case 'tools/list':
          result = { tools: getToolList() };
          break;

        case 'tools/call':
          result = await callTool(params);
          break;

        default:
          isError = true;
          result = { code: -32601, message: `Unknown: ${method}` };
      }

      return new Response(JSON.stringify({
        jsonrpc: '2.0', id, ...(isError ? { error: result } : { result }),
      }), {
        headers: { 'Content-Type': 'application/json',
          'Access-Control-Allow-Origin': '*' },
      });
    }

    return new Response('Not Found', { status: 404 });
  },
};

Step 3: Configure wrangler.toml

name = "my-mcp-server"
main = "worker.mjs"
compatibility_date = "2025-06-18"
workers_dev = true

Step 4: Deploy

npx wrangler login    # Opens browser for auth
npx wrangler deploy  # Deploy to edge network

That's it! Your MCP server is now live at https://my-mcp-server.your-subdomain.workers.dev.

Publishing to MCP Registry

Once your Worker is deployed, you can register it in the official MCP Registry using the Remote mode:

1. Create server.json

{
  "$schema": "https://modelcontextprotocol.io/schema/2025-03-26/server.json",
  "name": "io.github.demo112/try-make-mcp",
  "description": "MCP Factory — Multi-tool MCP server on Cloudflare edge",
  "version": "1.0.0",
  "remotes": [
    {
      "url": "https://try-make-mcp.xxx.workers.dev/mcp",
      "transport": "streamable-http"
    }
  ]
}

2. Add to your GitHub repo

Place server.json in the root of your GitHub repository. The MCP Registry crawler will discover it automatically.

3. Use mcp-publisher (alternative)

brew install mcp-publisher
mcp-publisher login github    # Device flow, 1 minute
mcp-publisher publish          # Done!

Pro tip: Both remotes and packages can coexist in server.json. Clients can choose their preferred installation method. This means you can offer both a Cloudflare Workers remote endpoint AND a PyPI package from the same listing.

Best Practices for Production

1. Keep It Stateless

Workers are stateless by design. Store session data in KV, D1, or Durable Objects if needed. For most MCP tools, pure compute is sufficient.

2. Handle MCP Protocol Correctly

  • Always respond to initialize with proper protocolVersion and capabilities
  • Return 204 No Content for notifications (no id)
  • Include proper JSON-RPC error objects for invalid methods

3. Set Proper CORS Headers

MCP clients like Claude Desktop, Cursor, and VS Code make cross-origin requests. Always include Access-Control-Allow-Origin: *.

4. Monitor Usage

Cloudflare provides real-time analytics at dash.cloudflare.com. Monitor request count, error rates, and CPU usage.

5. Version Your Tools

Include version in serverInfo. When adding or removing tools, increment the version and update server.json.
