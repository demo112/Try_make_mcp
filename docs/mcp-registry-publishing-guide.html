Two Publishing Paths: Packages vs Remote

📦 Package Mode

Publish to PyPI/npm/Docker first, then register metadata. The Registry verifies ownership by checking your package.

Good for: CLI tools, local servers

Requires: Package registry account

🌐 Remote Mode

Point to a public HTTP endpoint. No package registry needed. The server must be publicly accessible.

Good for: Cloud services, SaaS APIs

Requires: Public HTTP endpoint

💡 You can use BOTH modes simultaneously! server.json supports both packages and remotes fields. MCP clients can choose their preferred installation method.

Path A: Remote Mode (Recommended)

This is the most popular approach — 64% of Registry servers use Remote mode exclusively, with no packages at all.

Step 1: Deploy Your MCP Server

Deploy your MCP server to any public URL. Popular free hosting options include:

Step 2: Create server.json

{
  "$schema": "https://static.modelcontextprotocol.io/schemas/2025-12-11/server.schema.json",
  "name": "io.github.your-username/your-server",
  "title": "Your MCP Server",
  "description": "A brief description (max 100 chars)",
  "version": "1.0.0",
  "repository": {
    "url": "https://github.com/your-username/your-repo",
    "source": "github"
  },
  "remotes": [
    {
      "type": "streamable-http",
      "url": "https://your-server.example.com/mcp"
    }
  ]
}

Step 3: Install mcp-publisher

# macOS / Linux (binary download)
curl -L "https://github.com/modelcontextprotocol/registry/releases/latest/download/mcp-publisher_$(uname -s | tr '[:upper:]' '[:lower:]')_$(uname -m | sed 's/x86_64/amd64/;s/aarch64/arm64/').tar.gz" | tar xz mcp-publisher && sudo mv mcp-publisher /usr/local/bin/

# Or via Homebrew
brew install mcp-publisher

Step 4: Authenticate (1 minute)

mcp-publisher login github

This initiates a GitHub Device Flow:

To authenticate, please:
1. Go to: https://github.com/login/device
2. Enter code: ABCD-1234
3. Authorize this application
Waiting for authorization...
✓ Successfully logged in

Step 5: Publish!

mcp-publisher publish

That's it! Your MCP server is now in the official Registry.

Path B: Package Mode (PyPI)

If you have a Python MCP server, you can publish via PyPI.

Step 1: Add verification to your package

Include this comment in your README.md (it becomes the PyPI description):

<!-- mcp-name: io.github.your-username/your-server -->

Step 2: Create pyproject.toml

[project]
name = "your-server"
version = "1.0.0"
description = "Your MCP server"
readme = "README.md"
license = {text = "MIT"}
requires-python = ">=3.9"
dependencies = ["mcp"]

[project.scripts]
your-server = "your_package:main"

Step 3: Build and publish to PyPI

python -m build
twine upload dist/*

Step 4: Authenticate and publish to MCP Registry

mcp-publisher login github
mcp-publisher publish
⚠️ PyPI Verification The Registry will fetch your package from PyPI and check that the README contains the <!-- mcp-name: ... --> comment matching your server name in server.json.

Authentication Methods

GitHub OAuth (Easiest)

Uses Device Flow — no API tokens, no PATs needed.

  • Name format: io.github.username/*
  • One command: mcp-publisher login github
  • Opens browser for authorization

DNS TXT Record (Custom Domain)

For organizations with custom domains.

  • Name format: com.example/*
  • Requires DNS TXT record on apex domain
  • Supports Ed25519 and ECDSA P-384 keys

Best Practices

  1. Use streamable-http transport — it's the modern standard and works with all major MCP clients.
  2. Support both transports — include both streamable-http and sse URLs in your remotes for maximum compatibility.
  3. Keep descriptions under 100 characters — the Registry enforces this limit.
  4. Use semantic versioning — once published, version metadata is immutable.
  5. Include required headers — if your server needs API keys, document them in the headers array.
  6. Set up CI/CD — use GitHub Actions to auto-publish when you push new versions.
  7. Test your remote endpoint — ensure it's publicly accessible and responds to MCP initialization requests.
  8. Add a websiteUrl — link to documentation for discoverability.

Frequently Asked Questions

Do I need both packages and remotes?

No. You can use either or both. 64% of servers use remotes-only with no packages.

Can I publish without PyPI/npm?

Yes! Remote mode requires no package registry at all. Just a public HTTP endpoint and GitHub authentication.

Does the Registry validate my remote endpoint?

Yes — your remote server must be publicly accessible at the specified URL and respond to MCP protocol requests.

Can I update my server metadata?

Yes — submit a new server.json with a unique version string. Published versions are immutable (like npm).

Can I delete my server?

Yes — use mcp-publisher status --status deleted. The metadata is preserved but hidden from listings.

What authentication do I need?

GitHub OAuth (Device Flow) is the easiest — just one command and a 1-minute browser authorization.
