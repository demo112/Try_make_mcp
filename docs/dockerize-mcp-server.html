Dockerize MCP Server: Complete Production Deployment Guide (2026)

📅 May 2026 · ⏱️ 10 min read · 🐳 Docker
Docker DevOps Production

You've built your MCP server. Now you need to deploy it reliably — across your team, in CI/CD, and eventually to production. Docker is the answer. This guide covers everything from a minimal Dockerfile to production-grade patterns with health checks, multi-stage builds, and Docker Compose orchestration.

Prerequisites: Docker 20.10+ installed. Basic understanding of MCP servers. If you haven't built an MCP server yet, start with the 5-minute MCP Factory tutorial.

📦 Minimal Dockerfile

The simplest Docker setup for an MCP server. Works for most Python-based MCP servers:

Dockerfile (single-stage)
FROM python:3.11-slim

WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    tesseract-ocr \
    poppler-utils \
    && rm -rf /var/lib/apt/lists/*

# Copy requirements first (for layer caching)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Default command (can be overridden)
CMD ["python", "src/mcp_factory/run.py"]

🚀 Multi-Stage Build (Production Optimized)

Multi-stage builds reduce image size by 60-80% by separating build-time and runtime dependencies:

FROM python:3.11-slim AS builder

WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt


# Runtime stage
FROM python:3.11-slim

# Install only runtime system deps
RUN apt-get update && apt-get install -y --no-install-recommends \
    tesseract-ocr \
    poppler-utils \
    && rm -rf /var/lib/apt/lists/* \
    && useradd --create-home --shell /bin/bash mcpuser

WORKDIR /app

# Copy installed packages from builder
COPY --from=builder /root/.local /home/mcpuser/.local
ENV PATH=/home/mcpuser/.local/bin:$PATH

# Copy only runtime code
COPY --chown=mcpuser:mcpuser . .

USER mcpuser

CMD ["python", "src/mcp_factory/run.py"]

Result: ~150MB vs ~900MB for single-stage build.

🏥 Health Checks

Health checks let Docker and orchestrators know when your MCP server is ready to serve traffic:

FROM python:3.11-slim

WORKDIR /app
COPY . .
RUN pip install --no-cache-dir -r requirements.txt

# Health check: ping the MCP server every 30s, timeout after 5s
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
    CMD python -c "import sys; sys.exit(0)" || exit 1

CMD ["python", "src/mcp_factory/run.py"]
Note: MCP servers use STDIO transport by default, making traditional HTTP health checks tricky. For HTTP-transport MCP servers, add a /health endpoint. For STDIO servers, use a startup-time health check or Kubernetes readiness probes.

🐙 Docker Compose for Local Development

Docker Compose lets you run your MCP server alongside Claude Desktop for local development:

version: '3.8'

services:
  mcp-factory:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: mcp-factory
    volumes:
      # Mount your local code for hot-reload during development
      - .:/app
      - ~/.config/claude:/home/mcpuser/.config/claude:ro
    environment:
      - PYTHONUNBUFFERED=1
      - LOG_LEVEL=DEBUG
    restart: unless-stopped

  # Example: PostgreSQL for RAG pipeline
  postgres:
    image: pgvector/pgvector:pg16
    container_name: mcp-pg
    environment:
      POSTGRES_PASSWORD: dev_password
      POSTGRES_DB: rag_db
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

☁️ Production Deployment Patterns

Pattern 1: Single Server (DigitalOcean/App Runner)

# Build and run directly
docker build -t mcp-factory:latest .
docker run -d \
  --name mcp-factory \
  -p 8000:8000 \
  -e PRODUCTION=true \
  -e API_KEYS="key1,key2" \
  --restart unless-stopped \
  --health-cmd="curl -f http://localhost:8000/health || exit 1" \
  --health-interval=30s \
  mcp-factory:latest

Pattern 2: Kubernetes Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: mcp-factory
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mcp-factory
  template:
    metadata:
      labels:
        app: mcp-factory
    spec:
      containers:
        - name: mcp-factory
          image: your-registry/mcp-factory:latest
          ports:
            - containerPort: 8000
          env:
            - name: PRODUCTION
              value: "true"
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 10
            periodSeconds: 30
          readinessProbe:
            httpGet:
              path: /ready
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 10
