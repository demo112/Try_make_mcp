MCP Testing Complete Guide 2026

📅 May 13, 2026 ⏱️ 12 min read 📖 Testing & Quality

Testing MCP (Model Context Protocol) servers is fundamentally different from testing traditional REST APIs. MCP uses JSON-RPC 2.0 over bidirectional transports (stdio, SSE, Streamable HTTP), supports multiple interaction patterns (tools, resources, prompts), and has unique failure modes like transport disconnection and payment-required errors. This guide covers every testing strategy you need — from unit tests to production CI/CD pipelines.

1. Why MCP Testing is Different

Traditional API testing sends HTTP requests and checks responses. MCP testing must account for:

  • Bidirectional communication — The server can send notifications to the client
  • Multiple transport types — stdio, Server-Sent Events (SSE), and Streamable HTTP each have different failure modes
  • JSON-RPC 2.0 protocol — Request IDs, error codes, and batch processing add complexity
  • Multiple interaction types — Tools, Resources, and Prompts each have different schemas
  • Session management — Initialization handshake, capability negotiation, and graceful shutdown
  • Payment & rate limiting — Error code -32042 requires specific handling
Key Insight: A tool might work perfectly in isolation but fail when called through the MCP protocol layer due to serialization issues, transport problems, or capability mismatches.

3. Unit Testing: Testing Individual Tools

Unit tests validate your tool's business logic independently from the MCP protocol. Test input validation, output formatting, and error handling.

Python Example with pytest

# test_tools.py
import pytest
from mcp_factory.tools.calculator import calculate

def test_calculator_addition():
    """Test basic arithmetic tool."""
    result = calculate(operation="add", a=10, b=5)
    assert result == 15

def test_calculator_division_by_zero():
    """Test error handling for division by zero."""
    with pytest.raises(ValueError, match="Cannot divide by zero"):
        calculate(operation="divide", a=10, b=0)

def test_calculator_invalid_operation():
    """Test handling of unsupported operations."""
    with pytest.raises(ValueError, match="Unsupported operation"):
        calculate(operation="modulo", a=10, b=3)

# Test input validation
def test_calculator_type_validation():
    """Test that non-numeric inputs are rejected."""
    with pytest.raises(TypeError):
        calculate(operation="add", a="hello", b=5)

# Test MCP tool schema compliance
def test_tool_returns_valid_structure():
    """Ensure tool output matches expected MCP content format."""
    result = calculate(operation="add", a=1, b=2)
    assert isinstance(result, (int, float))
    assert result == 3

TypeScript Example with Vitest

// tools/calculator.test.ts
import { describe, it, expect } from 'vitest';
import { calculate } from './calculator';

describe('Calculator Tool', () => {
  it('should add two numbers', () => {
    expect(calculate('add', 10, 5)).toBe(15);
  });

  it('should handle division by zero', () => {
    expect(() => calculate('divide', 10, 0)).toThrow('Cannot divide by zero');
  });

  it('should validate input types', () => {
    expect(() => calculate('add', 'hello' as any, 5)).toThrow();
  });

  it('should return numeric results', () => {
    const result = calculate('multiply', 3, 4);
    expect(typeof result).toBe('number');
  });
});
Best Practice: Design your tool functions to be pure (no side effects) so they're easy to unit test. Inject dependencies (HTTP clients, databases) rather than importing them directly.

4. Integration Testing: Testing the MCP Protocol Layer

Integration tests verify that your server correctly implements the MCP protocol — initialization, capability negotiation, tool listing, and tool invocation via JSON-RPC 2.0.

Python Integration Test

# test_mcp_integration.py
import asyncio
import json
import pytest

async def test_initialize_handshake():
    """Test MCP server initialization handshake."""
    from mcp_factory.server import create_server

    server = create_server()

    # Simulate client sending initialize request
    init_request = {
        "jsonrpc": "2.0",
        "id": 1,
        "method": "initialize",
        "params": {
            "protocolVersion": "2024-11-05",
            "capabilities": {},
            "clientInfo": {
                "name": "test-client",
                "version": "1.0.0"
            }
        }
    }

    response = await server.handle_request(json.dumps(init_request))
    response_data = json.loads(response)

    assert response_data["jsonrpc"] == "2.0"
    assert response_data["id"] == 1
    assert "result" in response_data
    assert response_data["result"]["protocolVersion"] == "2024-11-05"
    assert "serverInfo" in response_data["result"]

async def test_list_tools():
    """Test that server correctly lists available tools."""
    server = create_server()
    await server.initialize()

    list_tools_request = {
        "jsonrpc": "2.0",
        "id": 2,
        "method": "tools/list",
        "params": {}
    }

    response = await server.handle_request(json.dumps(list_tools_request))
    response_data = json.loads(response)

    assert "result" in response_data
    tools = response_data["result"]["tools"]
    assert len(tools) > 0

    # Validate tool schema
    for tool in tools:
        assert "name" in tool
        assert "description" in tool
        assert "inputSchema" in tool
        # inputSchema must be valid JSON Schema
        assert tool["inputSchema"]["type"] == "object"

async def test_call_tool():
    """Test invoking a tool through the MCP protocol."""
    server = create_server()
    await server.initialize()

    call_request = {
        "jsonrpc": "2.0",
        "id": 3,
        "method": "tools/call",
        "params": {
            "name": "calculate",
            "arguments": {"operation": "add", "a": 10, "b": 5}
        }
    }

    response = await server.handle_request(json.dumps(call_request))
    response_data = json.loads(response)

    assert "result" in response_data
    assert response_data["result"]["content"][0]["text"] == "15"

async def test_invalid_method():
    """Test that server returns proper error for invalid methods."""
    server = create_server()
    await server.initialize()

    invalid_request = {
        "jsonrpc": "2.0",
        "id": 4,
        "method": "tools/nonexistent",
        "params": {}
    }

    response = await server.handle_request(json.dumps(invalid_request))
    response_data = json.loads(response)

    assert "error" in response_data
    assert response_data["error"]["code"] == -32601  # Method not found

5. E2E Testing: Testing Complete Client-Server Flows

End-to-end tests launch your server as a subprocess and communicate via stdio — exactly how Claude Desktop or Cursor would.

# test_e2e.py
import subprocess
import json
import pytest

@pytest.fixture
def server_process():
    """Launch MCP server as subprocess (stdio transport)."""
    proc = subprocess.Popen(
        ['python', '-m', 'mcp_factory.server'],
        stdin=subprocess.PIPE,
        stdout=subprocess.PIPE,
        stderr=subprocess.PIPE,
        text=True
    )
    yield proc
    proc.terminate()
    proc.wait(timeout=5)

def send_request(proc, request):
    """Send a JSON-RPC request and read the response."""
    request_str = json.dumps(request)
    proc.stdin.write(f"Content-Length: {len(request_str)}\r\n\r\n{request_str}")
    proc.stdin.flush()

    # Read response header
    header = proc.stdout.readline().strip()
    content_length = int(header.split(': ')[1])

    # Read response body
    body = proc.stdout.read(content_length)
    return json.loads(body)

def test_full_workflow(server_process):
    """Test complete E2E workflow: init → list tools → call tool."""
    # 1. Initialize
    init_response = send_request(server_process, {
        "jsonrpc": "2.0",
        "id": 1,
        "method": "initialize",
        "params": {
            "protocolVersion": "2024-11-05",
            "capabilities": {},
            "clientInfo": {"name": "e2e-test", "version": "1.0.0"}
        }
    })
    assert init_response["result"]["protocolVersion"] == "2024-11-05"

    # 2. Send initialized notification
    send_request(server_process, {
        "jsonrpc": "2.0",
        "method": "notifications/initialized"
    })

    # 3. List tools
    list_response = send_request(server_process, {
        "jsonrpc": "2.0",
        "id": 2,
        "method": "tools/list",
        "params": {}
    })
    assert len(list_response["result"]["tools"]) > 0

    # 4. Call a tool
    call_response = send_request(server_process, {
        "jsonrpc": "2.0",
        "id": 3,
        "method": "tools/call",
        "params": {
            "name": "calculate",
            "arguments": {"operation": "multiply", "a": 6, "b": 7}
        }
    })
    assert call_response["result"]["content"][0]["text"] == "42"

6. Mock MCP Clients for Testing

Create a reusable mock client for rapid testing. The @modelcontextprotocol/sdk provides InMemoryTransport for this purpose.

// mock-client.ts
import { Client } from '@modelcontextprotocol/sdk/client/index.js';
import { InMemoryTransport } from '@modelcontextprotocol/sdk/inMemory.js';
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';

export async function createTestPair(serverSetup: (server: McpServer) => void) {
  const server = new McpServer({ name: 'test-server', version: '1.0.0' });
  serverSetup(server);

  const [clientTransport, serverTransport] =
    InMemoryTransport.createLinkedPair();

  const client = new Client({ name: 'test-client', version: '1.0.0' });
  const serverClient = new Client({ name: 'test-server', version: '1.0.0' });

  await Promise.all([
    client.connect(clientTransport),
    serverClient.connect(serverTransport)
  ]);

  return { client, server: serverClient, cleanup: async () => {
    await client.close();
    await serverClient.close();
  }};
}

// Usage in tests
import { describe, it, expect } from 'vitest';
import { createTestPair } from './mock-client';

describe('MCP Tools', () => {
  it('should work through mock transport', async () => {
    const { client, cleanup } = await createTestPair((server) => {
      server.tool('echo', { message: 'string' }, async ({ message }) => ({
        content: [{ type: 'text', text: message }]
      }));
    });

    const result = await client.callTool({
      name: 'echo',
      arguments: { message: 'hello mcp' }
    });

    expect(result.content[0].text).toBe('hello mcp');
    await cleanup();
  });
});

7. Transport-Level Testing

Each MCP transport type has unique failure modes. Test each one your server supports:

TransportFailure Modes to TestRecovery Strategy
stdio Process crash, stdin/stdout pipe break, buffer overflow Parent process restart, health check ping
SSE Connection drops, server restart, firewall timeout Auto-reconnect with exponential backoff
Streamable HTTP POST failures, missing session ID, network latency Session resumption, request retry 
# test_transport_resilience.py
import asyncio
import pytest

async def test_sse_reconnection():
    """Test that SSE client reconnects after server restart."""
    from mcp_factory.client.sse_client import SSEClient

    # Initial connection
    await client.connect()
    tools = await client.list_tools()
    assert len(tools) > 0

    # Simulate server restart
    client.disconnect()

    # Should reconnect automatically

    tools_after = await client.list_tools()

8. Testing MCP Responses and JSON-RPC Compliance

Every MCP response must comply with JSON-RPC 2.0. Use these assertions:

# test_jsonrpc_compliance.py
import json

def validate_jsonrpc_response(response_str):
    """Validate that a response complies with JSON-RPC 2.0."""
    response = json.loads(response_str)

    # Must have jsonrpc version
    assert response.get("jsonrpc") == "2.0", "Missing jsonrpc version"

    # Must have either 'result' or 'error', never both
    has_result = "result" in response
    has_error = "error" in response
    assert has_result != has_error, "Must have result OR error, not both/neither"

    # Must have id matching request
    assert "id" in response, "Missing response id"

    if has_error:
        error = response["error"]
        assert "code" in error, "Error must have code"
        assert "message" in error, "Error must have message"
        # Standard JSON-RPC error codes
        assert isinstance(error["code"], int)

    return True

def test_all_mcp_errors_use_correct_format():
    """Test that MCP-specific errors use the reserved -32000 to -32099 range."""
    mcp_errors = [
        (-32001, "Resource not found"),
        (-32002, "Tool not found"),
        (-32042, "Payment required"),  # MCP billing error
    ]

    for code, message in mcp_errors:
        response = {
            "jsonrpc": "2.0",
            "id": 1,
            "error": {"code": code, "message": message}
        }
        assert validate_jsonrpc_response(json.dumps(response))

11. Testing Checklist for Production MCP Servers

✅ Must-Have Tests

  • ✅ Initialization handshake — Server responds correctly to initialize with capabilities
  • ✅ Tool listing — tools/list returns valid tool definitions with JSON Schema
  • ✅ Tool invocation — Each tool works with valid and invalid arguments
  • ✅ Error responses — Proper JSON-RPC error codes for all failure modes
  • ✅ Input validation — Invalid inputs return errors, not crashes
  • ✅ Protocol version — Correct 2024-11-05 version in responses
  • ✅ Resource listing — resources/list returns valid URIs and descriptions
  • ✅ Prompt listing — prompts/list returns valid prompt definitions
  • ✅ Transport resilience — Server handles connection drops gracefully
  • ✅ Concurrency — Multiple simultaneous tool calls work correctly
  • ✅ Large payloads — Server handles large tool arguments without crashing
  • ✅ Timeout handling — Long-running tools handle client timeouts

❌ Common Anti-Patterns

  • ❌ Only testing business logic, never testing the MCP protocol layer
  • ❌ Testing with curl to HTTP endpoints (doesn't test stdio/SSE transports)
  • ❌ Not testing the initialized notification exchange
  • ❌ Ignoring JSON-RPC error code conventions (using HTTP-like 404, 500)
  • ❌ Not testing tool schema — invalid schemas cause client crashes
